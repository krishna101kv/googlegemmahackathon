📋Executive Summary

Stagecraft is a local-first speech coaching application built with Next.js and powered by Gemma 4 native audio understanding. It records user speeches, transcribes them, and provides real-time AI-driven coaching feedback using Toastmasters-standard rubrics. The application prioritizes privacy by running on local infrastructure (Ollama) while maintaining optional cloud fallback capabilities.

Key Features

🎤 Browser Recording

Real-time audio capture with visualization and quality checks

🤖 Dual Inference

Local Ollama + cloud Gemini fallback for reliability

📊 Objective Metrics

WPM, filler words, pace categorization, and more

📈 Progress Tracking

Historical session review with trends and analysis

🎯 Goal-Based Coaching

Personalized focus areas and drill recommendations

💾 Local Storage

SQLite database with WAV audio archiving

🔍System Overview

The system follows a clean pipeline from audio capture through AI evaluation to persistent storage:

User Records Speech ↓ Browser MediaRecorder (WebM/Ogg) ↓ Client-side WAV Conversion (16kHz mono) ↓ POST /api/analyze (base64 WAV payload) ↓ Server: Audio Preparation (normalization, amplification, trimming) ↓ Branch: Local Ollama OR Cloud Gemini ├→ Ollama: Transcribe + Evaluate (two-stage) └→ Gemini: Transcribe + Evaluate (unified) ↓ Structured Coaching Evaluation (Zod validated) ↓ Objective Metrics Computation ↓ SQLite Session + Goal Storage ↓ Return Coaching Data to UI ↓ Display Results & Progress Tracking

⚙️Technology Stack

Frontend

  • Framework: Next.js 16.2.10 (App Router, React 19.2.4)
  • Styling: Tailwind CSS v4 with PostCSS
  • Data Visualization: Recharts 3.9.2
  • Type System: TypeScript 5.x
  • Validation: Zod 4.4.3

Backend

  • Runtime: Node.js (Next.js ServerFunctions + API Routes)
  • Database: SQLite 3 with WAL (better-sqlite3 12.11.1)
  • Audio Processing: Web Audio API (browser) + WAV parsing (Node.js)
  • LLM Orchestration: Ollama + Google Generative AI SDK

Infrastructure

  • Local Inference: Ollama (gemma4:e4b or gemma4:12b)
  • Cloud Fallback: Google AI Studio (Gemini models)
  • Storage: Local filesystem (data/audio/) + SQLite

🧩Core Components

1. AudioRecorder Component

Browser Component

Captures audio from user's microphone via MediaRecorder API.

  • Automatic codec selection (WebM Opus)
  • Real-time elapsed time display
  • Error handling for permission denial
  • Resource cleanup on unmount

2. PracticePanel Component

Main UI Component

Orchestrates the main recording & submission UI.

  • Speech type selection (prepared, table_topics, evaluation, interview_pitch)
  • Optional coaching goal selection
  • Inference mode toggle (local/cloud/auto)
  • Real-time audio preview playback
  • Loading state with animations
  • Error display and retry logic

3. SessionReviewView Component

Results Display

Displays coaching feedback and objective metrics.

  • Transcript display with filler word highlighting
  • Rubric scores (1-5 scales)
  • Strengths, improvement areas, and drills
  • Historical comparison capabilities
  • Audio playback controls

4. ProgressDashboard Component

Analytics Component

Visualizes trends over time using Recharts.

  • Line chart: Overall scores over sessions
  • Bar chart: Rubric category performance
  • Statistics: Improvement rate, consistency metrics
  • Filters by goal or speech type

🔄Data Flow

The complete end-to-end flow from recording to feedback:

  1. User Interface: User selects speech type, goal, and mode
  2. Audio Capture: MediaRecorder captures WebM/Ogg blob
  3. Audio Conversion: Browser converts to 16kHz mono WAV
  4. Request Submission: POST to /api/analyze with base64 payload
  5. Server Validation: Verify RIFF header and payload size
  6. Audio Preprocessing: Parse chunks, mix to mono, amplify, trim
  7. LLM Inference: Branch to local Ollama or cloud Gemini
  8. Metrics Computation: Calculate WPM, filler rate, pace category
  9. Database Persistence: Save session and audio file
  10. Response to Client: Return coaching data as JSON
  11. UI Rendering: Display SessionReviewView with results

🔌API Endpoints

POST /api/analyze

Primary Endpoint

Main coaching endpoint. Accepts audio, processes it, and returns evaluation.

Request Body:
{
  audioBase64: string,        // WAV file as base64
  durationSeconds: number,    // 0.5-1200s
  speechType: string,         // 'prepared' | 'table_topics' | ...
  goal: string,               // 'reduce_fillers' | ...
  preferMode?: string         // 'local' | 'cloud'
}

Response (200 OK):
{
  session: SessionRecord,
  metrics: ObjectiveMetrics,
  reconcileNote: string,
  inferenceMode: string,
  audioMeta: { durationSeconds: number }
}

GET /api/sessions

List all sessions with pagination.

GET /api/sessions/:id

Retrieve single session with full details.

GET /api/audio/:id

Serve session audio file (audio/wav).

GET /api/goals

List all goals (active, completed, suggested).

POST /api/goals

Create or update a goal.

GET /api/progress

Aggregate metrics across sessions with trends.

GET /api/health

Liveness check for orchestration (K8s, load balancers).

🎵Audio Processing Pipeline

Stage 1: Browser-Side Conversion

MediaRecorder Blob (WebM/Ogg) → WAV Blob (16kHz mono, PCM-16)

  • Decode using AudioContext API
  • Resample to 16kHz
  • Mix to mono if needed
  • Encode as WAV with RIFF header

Stage 2: Server-Side Preparation

Base64 WAV → Normalized & Amplified WAV Buffer

  • Validate RIFF header
  • Parse fmt and data chunks
  • Mix to mono and measure peak amplitude
  • Amplify if quiet (< 55% of target)
  • Trim to 28 seconds max
  • Write normalized WAV file

Stage 3: Quality Validation

  • isUsableTranscript() — Check minimum word count (≥ 5 words)
  • isPromptContaminated() — Detect ASR instruction leakage
  • stripPromptContamination() — Clean output text

🧠Inference Engines

Local Inference (Ollama)

Default Mode

Supported Models: gemma4:e4b (default), gemma4:12b (ASR)

Two-Stage Process:

  1. Transcription: Convert audio to text
  2. Coaching Evaluation: Analyze transcript and generate feedback

Timeout: 180s (configurable)

Retry: 1 attempt on failure

Cloud Inference (Google Gemini)

Fallback Mode

Prerequisite: GOOGLE_API_KEY environment variable

Model: gemma-4-e4b-it

Unified Process:

Single API call combines transcription + coaching evaluation

🚀Deployment & Configuration

Environment Variables

# Ollama Configuration
OLLAMA_BASE_URL=http://127.0.0.1:11434
OLLAMA_MODEL=gemma4:latest
OLLAMA_ASR_MODEL=gemma4:12b

# Google AI Studio (optional)
GOOGLE_API_KEY=<your-api-key>

# Inference Mode
INFERENCE_MODE=local              # 'local' | 'cloud' | 'auto'
AI_TIMEOUT_MS=180000
AI_MAX_RETRIES=1

Local Development

  1. Start Ollama: ollama serve
  2. Pull model: ollama pull gemma4:latest
  3. Install dependencies: npm install
  4. Copy environment: cp .env.example .env.local
  5. Run dev server: npm run dev
  6. Open: http://localhost:3000

⚠️Error Handling & Recovery

Common Errors

Error Cause Recovery
"Microphone access required" User denied permission Enable mic in browser settings
"Audio is not RIFF/WAV" Unsupported codec Retry with different codec
"model not found" Ollama model not installed ollama pull gemma4:12b
"Connection refused" Ollama not running Start Ollama: ollama serve
"Timeout" Inference too slow Increase timeout or use chunked ASR

Graceful Degradation

If local inference fails and cloud fallback is enabled, the system automatically retries with Google Gemini API.

⚡Performance Considerations

Bottlenecks

  1. LLM Inference (dominant): 10-40s per session
  2. Audio Processing: < 500ms (browser) + < 100ms (server)
  3. Database Operations: < 10ms for most queries

Optimization Strategies

  • Chunked ASR: For audio > 6s, split into 8s overlapping segments
  • Database Indexing: Add indexes on frequently queried columns
  • Frontend Optimization: Lazy load charts, paginate history (50 per page)
  • Caching: Cache transcripts for identical audio hashes (future)

🔒Security & Privacy

Privacy by Design

  • Local-First: Audio never leaves user's machine by default
  • Local Storage: Ollama runs locally, no cloud transmission required
  • Optional Cloud: Explicit opt-in with user-managed API keys

Input Validation

  • All requests validated with Zod schemas
  • Audio buffer size checks (100 bytes min, 50MB max)
  • Duration validation (0.5s - 20min)

Output Validation

  • LLM outputs validated against strict schemas
  • Rubric scores clamped to valid ranges (1-5)
  • Transcripts verified as verbatim (not rewritten)

Future Enhancements

  • Multi-user authentication (OAuth/SAML)
  • AES-256 encryption for audio files at rest
  • Audit logging and data access tracking
  • Rate limiting and DDoS protection
